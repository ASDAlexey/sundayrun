# Сокращение числа HTTP-запросов к `sundayrun.db`

> Статус: анализ + первый инкремент реализован (материализация `overallStats`).
> Автор черновика: агент, ночь 2026-07-19. Утром — на ревью.

## 1. Симптом

Главная страница (`localhost:4200/`) делает **~150 HTTP-запросов** (в DevTools видно
`60 / 159 requests`, все — `GET sundayrun.db`, статус `206 Partial Content`, размеры от 4 КБ
до 262 КБ). Это на странице, которой по смыслу нужно всего ничего: инфа о забеге, счётчик
до старта, превью 4 последних забегов и пара сводных чисел.

## 2. Почему так — механика `sqlite-wasm-http`

БД читается не целиком, а **постранично по HTTP range-запросам** (отсюда `206`). Настройки
(`src/app/github/protocol-db.service.constant.ts`):

- `PROTOCOL_DB_WORKER_COUNT = 1` — один воркер, запросы идут по одному;
- `maxPageSize: 4096` = `PRAGMA page_size` БД → **один range-запрос = одна страница 4 КБ**.

БД сейчас **848 КБ ≈ 212 страниц**. Каждый SQL-запрос идёт по B-tree и подтягивает по HTTP
**каждую страницу, которой касается** (страницы индекса + страницы таблиц), по одной за раз.
Значит число запросов ≈ числу страниц, которые трогают запросы страницы. Полный скан большой
таблицы = десятки-сотни range-запросов.

## 3. Кто именно генерит запросы на главной

`HomePage` в конструкторе запускает (`src/app/features/home/home-page.ts`):

| Загрузка                          | Запрос                          | Стоимость                                                         |
| --------------------------------- | ------------------------------- | ----------------------------------------------------------------- |
| `loadOverallStats()`              | `selectOverallStats`            | **очень дорого — см. ниже**                                       |
| `#loadLatest()`                   | `selectArchiveEvents(limit)`    | дёшево (events маленькая, есть индекс `events_date_iso`, `LIMIT`) |
| `siteMeta.load()`                 | чтение `meta`                   | дёшево (keyed по PK)                                              |
| `#loadSelf()` (если выбран атлет) | `loadRecord` + `loadEventSlugs` | средне                                                            |

### Главный виновник — `selectOverallStats`

`src/app/github/protocol-db-queries.ts:337`. Внутри — **три прохода по самой большой таблице
`runs` c join к `athletes`**:

1. `count()` + `countDistinct(athleteKey)` по всей `runs`;
2. `selectMedianTimeMs(male)` — `count` по фильтру **и `ORDER BY time_ms LIMIT/OFFSET`**;
3. `selectMedianTimeMs(female)` — то же ещё раз.

Индекса по `runs.time_ms` нет → `ORDER BY time_ms` **сортирует всю отфильтрованную таблицу**,
то есть читает почти все страницы `runs` + связанные страницы `athletes`. И так дважды (М/Ж).
Плюс count-distinct. Итог: `runs` фактически вычитывается 3+ раза постранично → **~сотня
range-запросов из одной этой загрузки**. Это ровно тот шторм, что на скриншоте.

## 4. Идея архитектуры — материализованные сводки

Ключевой принцип (ровно то, о чём просил пользователь):

> хранить общую/сводную информацию заранее, а детали подгружать лениво там, где они реально нужны.

БД пересобирается **на каждую публикацию** (`syncDbToState` в `protocol-db-write.ts`), и там уже
есть паттерн предрасчёта в транзакции: `recomputeStoredNotes`, `recomputeEventSummaryCounts`.
И есть kv-таблица `meta`. Значит: **считаем агрегаты один раз при записи и кладём в `meta`
(или в отдельную summary-таблицу), а страницы читают одну крохотную строку** вместо full-scan.

Для `overallStats` вообще не нужен SQL-агрегат: в write-пути под рукой полная `target.history`,
а `computeOverallStats(history)` (`core/history/overall-stats.ts`) уже считает ровно
`OverallStats`. Достаточно сохранить его JSON в `meta`.

Эффект: `loadOverallStats` с ~сотни range-запросов → **1–2** (keyed-lookup по PK `meta`).

## 5. Дорожная карта (по убыванию отдачи)

1. **[СДЕЛАНО] `overallStats` → `meta`.** Главная: 159 → 127 запросов. Реализовано, см. §6.
   Фолбэк на старый агрегат, если ключа нет (БД опубликована до изменения).
   1a. **[СДЕЛАНО] `trustBaked` — не перезапрашивать запечённое.** `loadWithTransfer.refresh()`
   применял baked-значение из TransferState И **всегда** дёргал `load()` заново в браузере.
   Но БД бандлится в тот же Pages-деплой, что и пререндер HTML → baked-значение уже актуально
   для этого деплоя, ре-фетч возвращает те же байты. Единственное, что теряем — редкий кейс
   устаревшего браузер-кэша HTML через деплой (данные меняются раз в неделю, UI и так пишет
   «может обновляться с задержкой»). Опция `trustBaked` пропускает ре-фетч при наличии baked;
   включена для трёх загрузок главной (превью/мета/тоталы). Карточка «выбери себя» остаётся
   живой — она браузер-only и никогда не пререндерится. Ожидаемо убирает основную массу
   оставшихся range-запросов главной.
2. **[СДЕЛАНО] Records / Итоги / детальный протокол — пререндер + `trustBaked`.** См. §7.
3. **Покрывающие индексы** для оставшихся live-запросов, которые нельзя предрассчитать:
   `runs(distance_km, time_ms)` (медианы-фолбэк, соперники), `results(slug, idx)` уже PK.
4. **Тюнинг транспорта:** проверить `page_size` 8–16 КБ (меньше страниц ↔ больше байт на
   запрос) и возможный prefetch/coalescing в `sqlite-wasm-http`. Мерить, не гадать.
5. **Снимок для above-the-fold:** пререндер уже печёт превью в статический HTML через
   TransferState. Рассмотреть отказ от live-refetch сводных чисел на главной (они меняются раз
   в неделю) — брать из TransferState/крохотного `home.json`, а БД трогать только по интеракции.

## 6. Что уже сделано в этом заходе (инкремент 1)

- `PROTOCOL_DB_META_OVERALL_STATS_KEY = 'overallStats'` — новый ключ `meta`.
- `storeOverallStats(db, history)` (`core/sqlite/protocol-db-overall-stats.ts`) — считает
  `computeOverallStats` и upsert-ит JSON в `meta`; вызывается в транзакции `syncDbToState`
  рядом с `recomputeEventSummaryCounts`.
- `selectStoredOverallStats(db)` в `protocol-db-queries.ts` — keyed-чтение `meta`; parse JSON.
- `selectOverallStats` теперь: сперва пробует сохранённую сводку, при отсутствии — старый
  агрегат (фолбэк для БД, опубликованных до изменения). Схема БД не меняется (kv-ключ), бампа
  версии не требуется; ключ появится при следующей публикации (CI пересобирает БД на `data/**`).
- `scripts/backfill-overall-stats.ts` — одноразовый backfill (как прежние `backfill-*.ts`):
  читает rollup из БД и кладёт `computeOverallStats(history)` в `meta` — ровно то, что записала
  бы следующая публикация. Идемпотентен, остальные таблицы (включая канарейку) не трогает.

## 7. Инкремент 3 — Records / Итоги / детальный протокол через пререндер + `trustBaked`

Эти три страницы **не** пекли данные и всё грузили в браузере:

- `records` — route был `Prerender`, но `#load` под `if (isPlatformBrowser)` → HTML с «loading».
- `races/:slug` и `year` — `RenderMode.Client` (в `app.routes.server.ts` их не было) → пустой shell.

«Та же правка», что на главной, потребовала сначала **сделать данные baked**:

1. **Общий примитив.** `transfer-load.ts`: добавлен `createTransferLoader()` — захватывает инъекции
   один раз (конструктор) и возвращает загрузчик, вызываемый многократно (на каждый route-param).
   `loadWithTransfer` теперь тонкая обёртка над ним. `trustBaked` работает как в §1a.
2. **records** — `#load` разбит на `#loadData()` (bundle 5 датасетов) + `#applyData()`, обёрнут в
   `loadWithTransfer({ trustBaked: true })`. Убран `isPlatformBrowser`-гард → грузит и на сервере
   (пререндер через node-БД, без HTTP), печёт в `records.data`. Bump-chart сезонов остался ленивым.
3. **races/:slug** — реактивный по slug: `createTransferLoader` + ключ `race.view.<slug>`,
   `#resolveState` → `#applyState(slug, …)` со stale-guard. `trustBaked`.
4. **year / year/:year** — аналогично, ключ `year.review.<year>` (`latest` для безпарамного `/year`).
5. **Серверные маршруты.** `app.routes.server.ts`: `races/:slug` и `year/:year` переведены в
   `Prerender` с `getPrerenderParams` — перечисляют slug'и/годы из node-БД (`inject()` в build-
   контексте). Добавлен безпарамный `year`.

**Проверка сборки:** `bun run build` → **Prerendered 264 static routes** (было ~6), 7.4 с. HTML
детальной несёт `ng-state` с `race.view.<slug>`, year — `year.review.<latest|год>`, records —
`records.data`. Браузер на пререндеренном URL применяет baked и **пропускает БД-запросы**.

**Стоимость:** вывод пререндера ~+31 МБ (249 протоколов × ~125 КБ, gzip ~30 КБ) — это CI-артефакт,
не репозиторий. Клиент-навигация на неотпечатанный в сессии URL честно грузит вживую (baked нет).

**Тесты:** home/records/race/year — 38/38 зелёные (co-run). Полный прогон падает на пред-
существующей проблеме резолва путей ассетов (wasm/xlsx) при мультифайловом запуске — не связано
с этими правками (те же падения на нетронутых `xlsx-reader`/`results.service`).

## 8. Страница атлета — пререндер НЕ подходит, нужна материализация

Пробовал тот же приём (`getPrerenderParams` по всем ключам + bake + `trustBaked`). Сборка прошла
(**513 маршрутов**), но **замер вскрыл нежизнеспособность**:

- одна страница атлета = **532 КБ** (gzip 56 КБ), папка атлетов = **129 МБ**, весь вывод = 175 МБ;
- причина: страница печёт **глобальные** агрегаты (`loadLegendFinishes` = ВСЕ забеги архива,
  `yearBests`, `winnerEvents`, погода) — одинаковый контент дублируется в каждую из 249 страниц.

**Масштабирование:** дублирование растёт как `атлеты × вся_история` → через ~год-два ~0.5–1 ГБ,
через 10 лет — несколько ГБ (превысит лимит GitHub Pages). Плюс сама страница растёт со всей
историей (через 10 лет ~2 МБ). Поэтому **пререндер атлетов откачен** (снова client-render, как был).
Также при пререндере всплыл SSR-баг: `ProgressChart` зовёт `getComputedStyle` (canvas-only) —
для будущего пререндера его надо гейтить `isPlatformBrowser` (в откате это тоже снято).

**Правильный масштабируемый путь (выбран, TODO):** материализовать при записи в БД не сырые
глобальные массивы, а их компактные РЕЗУЛЬТАТЫ, и читать дёшево:

- легенда трассы — снапшот текущей доски (кто легенда + оконные счётчики), не все забеги;
- year-rank пороги, winner-times, погодные экстремумы — компактные сводки в `meta`/summary-таблице;
- пересчёт в транзакции `syncDbToState` рядом с `recomputeStoredNotes` (тот же паттерн, что
  `overallStats`), покрыть тестами — write-путь критичен (через него публикуются все данные).
  Тогда страница атлета (client или пререндер) читает несколько маленьких строк вместо full-scan —
  масштабируется линейно. Это отдельная аккуратная задача, не сделана в этом заходе.
