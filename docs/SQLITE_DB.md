# Идея: SQLite вместо монолитных JSON

Статус: ⬜ идея, не начато

## Требования

- **Данные общие для всех**: сайт читают многие, публикуют несколько админов. Хранилище — центральное (репозиторий + CDN), как сейчас. Локальные браузерные хранилища (IndexedDB, OPFS) источником данных быть не могут — они живут в конкретном браузере; допустимы только как кэш.
- **Мгновенное обновление у всех**: после публикации протокола любой посетитель при загрузке страницы должен видеть свежие данные, без ожидания протухания CDN-кэша.
- **Ноль расходов навсегда**: проект — open source, бесплатное беговое сообщество и платным не станет. Никаких карт, платёжных аккаунтов и «пока бесплатных» тарифов, которые могут измениться. Данные открыты и лежат рядом с кодом — любой может форкнуть.

## Проблема

Данные архива — статические JSON в репозитории, раздаются через jsDelivr:

- `data/index.json` (~130K) — каталог 250 событий, грузится целиком на любой странице архива
- `data/athletes.json` (~416K, ~20K строк) — вся история спортсменов, грузится целиком ради одной страницы атлета или рекордов
- файлы монолитные и растут с каждым событием; ни фильтрации, ни пагинации
- свежесть держится на purge jsDelivr + `no-cache` ревалидации — работает, но purge ненадёжен (race с edge-кэшами)

Публикация протокола (`src/app/core/github/publish-event.ts`) — атомарный коммит 5 файлов через GitHub API, при этом `index.json` и `athletes.json` каждый раз переписываются целиком.

## Идея

Положить в репозиторий один файл **`data/protocol.db`** (SQLite) и выполнять SQL-запросы прямо из браузера через HTTP Range-запросы: браузер скачивает **только нужные страницы БД** (~килобайты), а не весь файл. Точечный запрос к базе в сотни мегабайт стоит ~1 KB трафика — архив может расти годами без деградации. Хранилище остаётся центральным и общим — меняется только формат файла и способ чтения.

Библиотеки:

- [`sqlite-wasm-http`](https://github.com/mmomtchev/sqlite-wasm-http) — HTTP VFS на официальном [SQLite WASM](https://sqlite.org/wasm), рекомендуемый вариант
- [`sql.js-httpvfs`](https://github.com/phiresky/sql.js-httpvfs) — первоисточник паттерна (phiresky), maintenance mode

Чтение — read-only (Range-запросы), запись — как сейчас: собрать обновлённый `.db` в памяти браузера при публикации (официальный sqlite-wasm экспортирует базу в `Uint8Array`) и закоммитить через GitHub API.

## Как это работает по шагам

Сервера нет и не появляется. `.db` — это обычный файл в репозитории, как сейчас JSON. Вся «магия» — в том, _как_ браузер его читает.

### Чтение: любой посетитель сайта

1. Сайт грузит `data/version.json` (~50 байт) → узнаёт актуальную версию данных (SHA коммита).
2. Библиотека (`sqlite-wasm-http`) открывает «виртуальное подключение» к `.../@<sha>/data/protocol.db`. **Весь файл при этом не скачивается** — браузер лишь узнаёт его размер (один HEAD-запрос).
3. Посетитель открывает, например, страницу атлета → выполняется `SELECT * FROM runs WHERE athlete_key = '...'`. SQLite внутри браузера (WASM) вычисляет, какие именно байты файла нужны: сначала пара страниц индекса, по ним — страницы с данными этого атлета.
4. Каждая нужная страница — маленький HTTP-запрос `Range: bytes=X–Y` (~1–4KB). Страница атлета — это ~5–10 таких запросов, суммарно десятки KB **вместо скачивания всего `athletes.json` на 416KB**.
5. CDN отдаёт эти куски и кэширует их навсегда — URL версионный, содержимое по нему никогда не меняется.

Посетителю ничего не устанавливается и не хранится у него постоянно: БД лежит на CDN, браузер каждый раз тянет только нужные байты (плюс обычный HTTP-кэш браузера делает повторные открытия мгновенными).

### Как обновление доходит до всех сразу

1. После публикации меняется единственный файл — `version.json` (в нём новый SHA).
2. Он читается с `no-cache`-ревалидацией: при каждой загрузке страницы браузер спрашивает CDN «файл изменился?» — это копеечный запрос (обычно ответ 304, «нет»).
3. Как только там новый SHA — **все** запросы к данным автоматически идут по новому URL (`@<новый sha>/...`), т.е. к новой версии БД. Никого не надо «оповещать» и ничего не надо чистить: старые куски в кэшах лежат под старыми URL и просто никем больше не запрашиваются.
4. Итог: каждый посетитель видит свежие данные при первой же загрузке/обновлении страницы после публикации. Для уже открытых вкладок — опциональная перепроверка `version.json` при возврате на вкладку.

### Запись: админ публикует протокол

Запись — это не «запрос к базе», а публикация нового файла `.db`, ровно как сейчас публикуются JSON:

1. Только админ с GitHub-токеном (как сейчас на `/admin`).
2. Его браузер скачивает текущий `protocol.db` **целиком** (~500KB — единственное место, где файл качается весь, и это делает только админ раз в неделю).
3. Открывает его в памяти (SQLite WASM) и применяет событие обычным SQL-ом: `INSERT` события, `DELETE`+`INSERT` результатов, `UPDATE` атлетов и их забегов.
4. Экспортирует базу обратно в байты (`Uint8Array` → base64) и коммитит через GitHub API — в тот же атомарный коммит, что xlsx/PDF/JSON.
5. Вторым мини-коммитом обновляет `version.json` → с этого момента все читатели (п. «Как обновление доходит до всех») получают новую версию.

```
Админ                                Читатели (все)
─────                                ──────────────
xlsx → парсинг → SQL в памяти        version.json (50 байт) → SHA
      ↓ export .db                          ↓
коммит в GitHub ──→ jsDelivr CDN ──→ Range-куски БД по @SHA (~KB)
      ↓                                     ↓
коммит version.json                  свежие данные у всех
```

## Мгновенная синхронизация: версионный указатель

Ключевой механизм под требование «у всех всё сразу обновляется» (нужен и для JSON, и обязателен для `.db`):

1. Крошечный файл-указатель `data/version.json` (`{ "sha": "<commit>" }`, ~50 байт) — единственный файл, который читается с `no-cache` ревалидацией (дёшево, обычно 304).
2. Все данные грузятся по **SHA-pinned URL** jsDelivr: `cdn.jsdelivr.net/gh/<owner>/<repo>@<sha>/data/...` — контент иммутабельный, кэшируется вечно, никаких purge и гонок с edge-кэшами.
3. Публикация: основной коммит данных → из ответа GitHub API берём его SHA → второй мини-коммит обновляет `version.json` (purge только для него, вместо нынешнего purge пяти путей).
4. Для SQLite это критично вдвойне: все Range-куски одной сессии гарантированно читаются из **одной версии** БД — исключена порча данных, когда файл сменился посреди чтения.
5. Свежесть при загрузке страницы — гарантирована. Для уже открытых вкладок (опционально): перепроверка указателя на `visibilitychange`/фокусе вкладки.

Существующий `site-meta.json` — это настройки (время старта, объявление), указателем не является; `version.json` — отдельный файл.

## Архитектура (гибрид)

JSON остаётся источником истины (git-diff, отладка, `regenerate-index.ts`), `.db` — производный артефакт для быстрых чтений.

### Схема БД

- `events` — из `index.json` (slug, dateIso, number, city, park, счётчики, лучшие времена)
- `results` — из по-событийных `results.json` (slug → строки протокола)
- `athletes` + `runs` — из `athletes.json` (атлет, его забеги, лучшие результаты)
- индексы: `athletes.key`, `runs.athlete_key`, `results.slug`, `events.dateIso`

### Запись (публикация)

В `publish-event.ts` шестым файлом коммитить обновлённый `data/protocol.db`:

1. скачать текущий `.db` (по SHA из `version.json` — консистентно)
2. применить событие SQL-ом (upsert event, replace results, обновить athletes/runs)
3. `export → Uint8Array → base64` → в общий атомарный коммит
4. вторым мини-коммитом обновить `version.json` + purge только его

Несколько админов: атомарный коммит через GitHub API уже даёт защиту — при параллельной публикации второй коммит упадёт на конфликте ref, нужно перечитать и повторить (сейчас это так же).

### Чтение

Переводить сервисы `src/app/github/*.service.ts` постепенно, начиная с самых тяжёлых:

1. страница атлета и рекорды (сейчас тянут весь `athletes.json`) → `SELECT` по ключу / top-N
2. список событий → `SELECT` с пагинацией вместо полного `index.json`
3. `results.json` можно не трогать — они и так маленькие (2–5K)

## Риски / проверить первым делом

- **Range-запросы на jsDelivr**: поддержка `Accept-Ranges` и корректных заголовков для бинарного файла. Fallback: `raw.githubusercontent.com` или сам GitHub Pages.
- WASM-ассеты (`.wasm` + worker) — добавить в сборку/`public/`.
- Бинарный файл в git растёт с каждым коммитом истории — следить за размером repo (сейчас `data/` уже 40MB из-за PDF, `.db` добавит ~500K на снапшот).
- Page size БД влияет на гранулярность скачивания — рекомендуемый 1024 байта для HTTP VFS.
- Публикация становится двумя коммитами (данные + указатель): короткое окно, когда указатель ещё старый — читатели просто видят предыдущую версию, ничего не ломается.

## Лимиты GitHub / jsDelivr и запас по размеру

Проверенные потолки (июль 2026):

| Лимит                   | Значение                                 | Комментарий                                                 |
| ----------------------- | ---------------------------------------- | ----------------------------------------------------------- |
| Файл в git              | 100 MiB жёсткий блок, warning с 50 MiB   | `protocol.db` дойдёт до этого не скоро                      |
| Blob через GitHub API   | 100 MB                                   | publish-flow коммитит base64-blob — хватает                 |
| Репозиторий             | рекомендация < 1 GB, настоятельно < 5 GB | жёсткого блока нет                                          |
| Файл на jsDelivr `/gh/` | **20 MB** (403 при превышении)           | главный потолок для `.db`; fallback — raw.githubusercontent |
| Git LFS                 | 1 GB хранилища / 1 GB трафика в месяц    | jsDelivr и raw **не отдают** LFS-контент — не использовать  |

Прогноз роста: `athletes.json` +~90 KB/год, PDF +~8 MB/год, история снапшотов `.db` +~50 MB/год → суммарно ~1 GB за 20 лет.

**Решение**: запаса в ~1 GB на 20 лет достаточно — отдельный репозиторий/orphan-ветку для `.db` не заводим. Если данные начнут расти быстрее прогноза (или GitHub напишет про размер) — адаптируемся тогда: orphan-ветка с force-push, пересборка `.db` из JSON или вынос истории.

## Шаг 0 (независимо полезен): версионный указатель + кэш

Механизм `version.json` + SHA-pinned URL можно внедрить **до** SQLite, прямо для текущих JSON — это уже закроет требование «мгновенно у всех» надёжнее нынешнего purge. Поверх него кэш распарсенных данных в IndexedDB (ключ — SHA) делает повторные визиты мгновенными и всегда консистентными: SHA совпал — берём из кэша, изменился — качаем новое.

## Альтернатива: облачная БД с фронта (бесплатно, доступно из РФ)

Если уходить от «данные в git» совсем: BaaS-платформы, к которым браузер обращается напрямую (без своего бэкенда). Бонус — «мгновенно у всех» решается само собой: нет CDN-кэша, читатели ходят прямо в БД.

| ---------------------------------------------------------------------------------- | ---------------------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| [Supabase](https://supabase.com)                                                   | Postgres BaaS                      | 500MB БД, 5GB трафика, 2 проекта, без карты     | Anon key в коде + RLS: чтение всем, запись только авторизованным админам (Supabase Auth)                                                    | Работает, ограничений не объявляли; оплата платных тарифов картами РФ — нет |
| [Turso](https://turso.tech)                                                        | SQLite (libSQL) в облаке, HTTP API | 5GB, 100 БД, 500M чтений строк/мес              | `@libsql/client/web` прямо из браузера; read-only токен в коде, write-токен — у админов (та же модель доверия, что сейчас с GitHub-токеном) | Явных блокировок нет                                                        |
| [Yandex Cloud](https://yandex.cloud/ru/docs/billing/concepts/serverless-free-tier) | YDB serverless + Object Storage    | 1M операций + 1GB YDB/мес; serverless free tier | Чисто с фронта нельзя — нужен API Gateway + Cloud Functions (тоже в free tier), т.е. мини-бэкенд                                            | Российский хостинг — самый надёжный по доступности, данные в РФ             |

| Cloudflare D1                                                                      | SQLite на edge                     | 5GB, 5M чтений/день                             | Только через Worker (не чисто с фронта)                                                                                                     | Работает                                                                    |

Оценка под требование «ноль расходов навсегда, читатели в России»:

- **Supabase / Turso** — технически хороши, но это VC-компании с «пока бесплатными» тарифами: лимиты и правила меняются, аккаунт может потребовать карту, данные уезжают из репозитория (лок-ин). Для проекта, который никогда не станет платным, — риск, а не решение.
- **Yandex Cloud** — требует платёжный аккаунт с картой даже для free tier → противоречит требованию, отпадает.
- **Firebase** — отпадает (риски Google в РФ).

**Вывод: git-модель остаётся основной.** GitHub бесплатен для open source навсегда, данные открыты и лежат рядом с кодом, любой может форкнуть. Идея SQLite-over-HTTP выше не добавляет ни копейки расходов.

## Страховка от блокировок: зеркало на российском git-хостинге

Единственный реальный риск git-модели для читателей в России — блокировка/замедление GitHub или jsDelivr. Закрывается бесплатным зеркалом на российском хостинге:

- [GitVerse](https://gitverse.ru) (СберТех) — бесплатно, вход по Sber ID, есть **GitVerse Pages** (статический хостинг `username.gitverse.page`)
- [GitFlic](https://gitflic.ru) — бесплатно для публичных проектов (до 4GB на репозиторий), тоже есть хостинг статических сайтов

Схема:

1. GitHub Action на push зеркалит репозиторий (`git push --mirror`) на GitVerse/GitFlic — автоматически, бесплатно.
2. Фронт получает список источников данных: jsDelivr → raw GitHub → зеркало в РФ. При недоступности первого — фолбэк на следующий.
3. При желании и сам сайт можно дублировать на GitVerse Pages — полная независимость от доступности GitHub для читателей.

Это сочетается со всеми идеями выше: и JSON, и `protocol.db`, и `version.json` — просто файлы, зеркалятся вместе с репозиторием.

## PoC (первый шаг для SQLite)

1. Скрипт `scripts/build-db.ts` (bun): собрать `protocol.db` из текущих JSON.
2. Закоммитить в ветку, из devtools выполнить запрос через `sqlite-wasm-http` к jsDelivr URL (SHA-pinned).
3. В Network tab убедиться, что идут Range-куски, а не полный файл.
4. Замерить время до данных на странице рекордов: полный `athletes.json` vs SQL top-N.
