# Backend: бесплатный хостинг + бэкап (Rust + Postgres)

План перехода с клиентского SQLite (sqlite-wasm-http) на серверный API
**Rust + PostgreSQL** с бесплатным хостингом, доступным из РФ, и защитой данных
через автобэкап наружу.

Статус: черновик плана. Код пишется отдельно.

---

## 1. Целевая архитектура

Фронт (Angular) перестаёт тянуть SQLite по HTTP range-запросам и ходит в обычное
REST API. Вся боль клиентского SQLite (range-запросы, патч размера под gzip
GitHub Pages, VFS) исчезает в момент появления серверного API — независимо от того,
какая БД внутри.

```
Angular SPA  ──HTTPS──>  Rust API  ──>  PostgreSQL
                          (axum)         (~20 МБ данных)
```

### Стек API

- **axum** — веб-фреймворк
- **sqlx** (Postgres, async, compile-time проверка запросов) + `sqlx migrate`
- **utoipa** + **utoipa-swagger-ui** — OpenAPI из `#[derive]`, Swagger UI на `/swagger-ui`

`Cargo.toml` (основное):

```toml
[dependencies]
axum = "0.8"
tokio = { version = "1", features = ["full"] }
sqlx = { version = "0.8", features = ["runtime-tokio", "postgres", "macros", "migrate", "chrono"] }
utoipa = { version = "5", features = ["axum_extras", "chrono"] }
utoipa-swagger-ui = { version = "8", features = ["axum"] }
serde = { version = "1", features = ["derive"] }
```

### Заметка про SQLite vs Postgres

Для 20 МБ read-mostly данных Postgres — избыточен (idle ест ~80–150 МБ RAM), и на
микро-хосте SQLite был бы легче. Выбран Postgres сознательно: нормальный SQL,
миграции, конкурентная запись на будущее. На хосте закладывать ≥1 ГБ RAM.

## 2. Хостинг

### Реальность «бесплатно + РФ + Postgres»

Надёжного «бесплатно навсегда» с persistent-сервисом **без иностранной карты** —
практически не существует. Всё по-настоящему бесплатное навсегда требует карту на
верификации. Fly.io / Render / Railway / Shuttle — иностранная карта + часть
блокируется/тормозит из РФ.

### Вариант A (выбранный): Oracle Cloud Always Free

Доступен, т.к. есть иностранная (EUR) виртуальная карта **«Плати по миру»**
(верификация — возвратный холд ~$1).

Что даёт бесплатно навсегда (не триал):

- **4 ARM-ядра Ampere A1 + 24 ГБ RAM** — можно резать на несколько VM
  (напр. 2 × 2 ядра/12 ГБ или 4 × 1 ядро/6 ГБ) → «несколько серверов в будущем»
- **+ 2 AMD micro-VM** (1 ГБ) тоже always-free
- **200 ГБ диска**, публичный IP

**Регион (home region):** Frankfurt / Amsterdam / Stockholm (~40–60 мс до РФ,
не блокируется). Выбирается один раз и не меняется.

**Сборка Rust:** под `aarch64` (A1 — это ARM, не x86). Multi-stage Dockerfile,
статик-бинарь musl.

Раскладка на одном A1 VM (Ubuntu 22.04, ARM64), docker-compose:

```
caddy      → авто-HTTPS (Let's Encrypt), проксирует на api
rust-api   → axum + sqlx + utoipa (Swagger на /swagger-ui)
postgres   → 20 МБ данных, том на block storage
```

### Подводные камни Oracle

1. **Карта на верификации.** Oracle строг к prepaid/виртуалкам — иногда режет.
   Пробовать; при отказе — план Б. Регистрация капризна по антифроду, бывает
   нужно 2–3 попытки.
2. **Reclaim простаивающих ARM-инстансов.** Лечится переключением аккаунта на
   **Pay As You Go** — в пределах free-лимитов денег не спишет, но reclamation
   отключается. Компромисс: PAYG возвращает зависимость от живой карты (см. §3).

### План Б (если карту не примут)

- **Amvera Cloud** — РФ PaaS, free-баланс на микро-апп, git-push, managed Postgres,
  оплата картой РФ.
- **Cloud.ru Evolution** — РФ, есть always-free тир на compute (проверить лимиты).
- **Дешёвый РФ-VPS** (Timeweb / Beget / RuVDS) ~150–250₽/мес — самый надёжный:
  один маленький VPS, docker-compose `api + postgres`, полный контроль.

## 3. Что если карта в будущем исчезнет — удалятся ли серверы?

**На чистом Always Free карта не списывается, поэтому её протухание само по себе НЕ
удаляет серверы.** Карта — пропуск на регистрацию (+ возвратный холд). Пока в
пределах бесплатного тира, биллинга нет → нет причины для блокировки.

| Режим аккаунта                                      | При мёртвой карте                                                                              |
| --------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **Pure Always Free** (не апгрейдил)                 | Списаний нет → карта не нужна → серверы живут. **НО** остаётся риск reclaim простаивающего ARM |
| **Pay As You Go** (апгрейд ради отключения reclaim) | Выход за free-лимиты + карта не платит → письма → grace (недели) → suspend → terminate         |

Ловушка: PAYG убирает reclaim, но возвращает зависимость от живой карты.

### Выбор режима

- «Не думать» → **остаться на Always Free**, держать инстанс не пустым (постоянная
  нагрузка / крон), чтобы не попасть под reclaim. Карта тогда почти не важна.
- Гарантия от reclaim → **PAYG**, но держать на карте живой валидный метод.

### Главная защита — переносимость (важнее гаданий про карту)

Серверы должны быть **одноразовыми и воспроизводимыми**. Данных ~20 МБ — исчезновение
провайдера = ~10 минут передеплоя, а не потеря всего.

1. **Инфра в git** — `docker-compose.yml` + Dockerfile + миграции. Новый сервер
   поднимается одной командой на любом провайдере.
2. **Автобэкап БД наружу** — см. §4.
3. **Не держать состояние только у одного free-хоста.** Тогда «несколько серверов»
   = просто `docker compose up` + восстановление одного дампа.

## 4. Автобэкап БД на Яндекс.Диск (крон, только при изменениях)

Яндекс.Диск поддерживает **WebDAV** — заливка одной `curl`, без лишних бинарей.

Пароль — **не основной, а пароль приложения**
(Яндекс ID → Безопасность → Пароли приложений). С 2FA основной не пустит.

### Способ А — по хешу дампа (без изменений в коде API)

```bash
#!/usr/bin/env bash
set -euo pipefail

pg_dump --no-owner --no-privileges -U app appdb | gzip > /tmp/db.sql.gz
NEW=$(sha256sum /tmp/db.sql.gz | cut -d' ' -f1)
OLD=$(cat /var/backup/last.sha256 2>/dev/null || true)

[[ "$NEW" == "$OLD" ]] && { echo "нет изменений, пропуск"; exit 0; }

curl -sf --user "$YANDEX_LOGIN:$YANDEX_APP_PASSWORD" \
  -T /tmp/db.sql.gz \
  "https://webdav.yandex.ru/backups/appdb-$(date +%F).sql.gz"

echo "$NEW" > /var/backup/last.sha256
```

Нюанс: `pg_dump` теоретически может дать разный байт-порядок строк при тех же данных
(после VACUUM/реорганизации) → изредка «ложный» бэкап. Для маленькой стабильной БД
почти не встречается.

### Способ Б — по маркеру записи (100% точно)

API — единственный писатель, поэтому бампает счётчик на каждый мутирующий запрос:

```sql
CREATE TABLE meta (key text primary key, val bigint);
-- в каждом POST/PUT/DELETE:  UPDATE meta SET val = val + 1 WHERE key = 'writes';
```

Скрипт сравнивает `SELECT val FROM meta WHERE key='writes'` с сохранённым числом —
дампит только при росте. Без ложных срабатываний, не зависит от капризов `pg_dump`.

**Рекомендация:** старт на способе А (ноль изменений в коде). Нужна идеальная
точность — добавить маркер Б (5 строк в API).

### Крон + ротация

```
0 3 * * * /opt/backup.sh >> /var/log/backup.log 2>&1
```

Ротация: держать последние ~14 дампов, старые удалять (WebDAV `DELETE` или
`rclone delete --min-age`). Альтернатива `curl` — `rclone` (backend «Yandex Disk»,
удобнее ротация и ретраи, нужен один раз `rclone config`).

Восстановление: `gunzip -c appdb-YYYY-MM-DD.sql.gz | psql -U app appdb`.
