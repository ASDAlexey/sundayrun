# SundayRUN — Воскресный парковый пробег

Бесплатное веб-приложение клуба «Воскресный парковый пробег» (Таганрог, КЛБ «Легенда»). Организатор загружает xlsx-экспорт приложения-секундомера → приложение определяет пол участников, считает места и авто-примечания → одним атомарным коммитом публикует результаты забега в каталог `data/` этого же репозитория (`sundayrun`). PDF-протокол по официальному образцу генерируется на лету в браузере (в мастере публикации и по кнопке «PDF» в архиве) — в репозитории он не хранится. Главная страница сайта — архив всех опубликованных забегов.

## Роли

- **Посетитель** — главная `/`: список забегов (свежий сверху) с датой, числом участников и кнопкой «PDF» (протокол генерируется на лету из опубликованных данных прямо в браузере). Архив читается с jsDelivr CDN (`data/index.json` репозитория `sundayrun`), поэтому после публикации список может обновиться с небольшой задержкой.
- **Организатор** — по неприметной ссылке «для организатора» открывает `/admin` и один раз вставляет GitHub-токен. После этого доступен мастер `/upload → /preview → /result`: загрузка xlsx, правка пола/примечаний и реквизитов события, предпросмотр PDF и «Опубликовать в архив» — один атомарный коммит (`source.xlsx`, `results.json` + обновлённые `index.json` и `athletes.json`) в `data/` репозитория `sundayrun`. Статистика атлетов (кол-во забегов, рекорды за всё время и по годам) при этом пересчитывается автоматически. Ошибочно загруженный забег можно удалить на `/admin` (блок «Удаление забега») — таким же атомарным коммитом файлы события удаляются, а `index.json`/`athletes.json` пересчитываются без его вклада.

## Стек

Angular 22 (standalone, zoneless, signals) · Bun · Vitest (покрытие 100%) · fflate (парсинг xlsx) · pdfmake + PT Serif (PDF с кириллицей) · GitHub Pages + GitHub API (хостинг и хранилище, всё бесплатно) · i18n: ru (основной) / en

## Команды

```bash
bun install                  # зависимости
bun start                    # dev-сервер (ru), http://localhost:4200
bun run start:en             # dev-сервер (en)
bun run test                 # юнит-тесты + покрытие
bun run check:code-quality   # prettier + eslint + stylelint
bun run build                # прод-сборка обеих локалей
bun run madge                # проверка циклических зависимостей
bun run generate-i18n        # извлечение i18n-сообщений → src/locale/messages.xlf
```

После `generate-i18n` новые trans-unit'ы нужно вручную перевести в `src/locale/messages.en.xlf` — сборка идёт с `i18nMissingTranslation: "error"` и упадёт на пропущенном переводе.

## Хранилище данных (`data/` в этом репозитории)

Код и опубликованные данные живут в одном публичном репозитории `ASDAlexey/sundayrun` (ветка `main`); все данные — под `data/`:

- `data/index.json` — список забегов (стартовое значение `{"schemaVersion":1,"events":[]}`)
- `data/athletes.json` — история участников (стартовое значение `{}`)
- `data/events/<дата>/` — файлы события (`source.xlsx`, `results.json`); PDF в репозитории не хранится, а генерируется на лету из `results.json`

Приложение пишет в `data/` само атомарными коммитами; вручную эти файлы не редактировать. CI пропускает деплой для пушей, меняющих только `data/**`.

## Токен организатора (fine-grained PAT)

Подробная пошаговая инструкция с ответами на частые вопросы: [docs/ADMIN_TOKEN.md](docs/ADMIN_TOKEN.md) (она же открывается по ссылке «Как создать токен» на `/admin`).

1. GitHub → Settings → Developer settings → Fine-grained personal access tokens.
2. Repository access: **Only select repositories → `sundayrun`**.
3. Permissions: **Contents → Read and write** (больше ничего).
4. Вставить токен на странице `/admin` — приложение проверит доступ и сохранит его в localStorage браузера. Токен и есть «роль админа»; сервера аутентификации нет.
5. Когда токен истечёт, публикация покажет баннер «Нет доступа к GitHub» со ссылкой на `/admin` для замены токена.

## Доимпорт старой истории

Старые забеги можно опубликовать задним числом — **строго от старых к новым**, чтобы история `athletes.json` копилась в хронологическом порядке:

1. Загрузить самый старый xlsx, на `/preview` заполнить дату и остальные реквизиты.
2. Нажать «Подставить примечания из истории» — примечания («Первое участие», «ЛР (было X)», «Лучший результат YYYY г.») считаются из уже опубликованной истории; при необходимости поправить вручную.
3. Сгенерировать PDF и опубликовать; повторить для следующего по дате файла.

Перепубликация той же даты идемпотентна: событие в `index.json`/`athletes.json` перезаписывается, дубликатов не появляется.

## Деплой (GitHub Pages)

В настройках репозитория приложения: **Settings → Pages → Source: GitHub Actions**. Workflow `.github/workflows/ci.yml` на каждый push в `main` (и по кнопке Run workflow) прогоняет линтеры и тесты, собирает обе локали с `--base-href /<repo>/`, добавляет корневой редирект на `/ru/` и SPA-fallback `404.html`, затем деплоит на Pages.

## Данные

Примеры входного Excel и целевого PDF — в `assets/`. Формат входа: экспорт приложения-секундомера (колонки Name, Total, Avg/lap, Avg/km, Lap 1, Lap 2). Опубликованные протоколы и история хранятся в этом же репозитории под `data/` (см. архитектуру).
